How to measure the cookbook — proofread every bolt before judging the machine

Four independent model families were asked, without seeing each other's answers, how to measure the cookbook's whole capability as a static, measurable score. They converged.

02Phase 1 — Proofread the writing

Observability first. Before you can say the cookbook works, you must be able to see, for every instruction, whether it was followed. This is the cheap, fast, iterable, static layer.

The instrument

A frozen registry: one instruction → one checker

Compile the cookbook into atomic instructions — like the numbered, single-action, verifiable steps of a real service manual. Each gets an immutable ID, its exact source text, a deterministic "does this even apply here?" predicate, the required evidence, a versioned checker (hashed, with golden pass/fail fixtures and mutation tests), and its possible fault codes.

  • Scripts do the checking, on frozen artifacts only — git state, files, processes, the peg, the final diff. Same artifacts in → identical result out. That is what makes the score static.
  • Checkers verify substance, not ritual, and are paired with a negation-invariant so an agent can't fake the gesture.
  • AI is bounded — used only where a script genuinely can't reach, restricted to yes/no classification on a fixed, frozen artifact, pinned + sampled, its verdict written to a committed label. Never in the live scoring path.

Keep the three questions apart (GPT). "Followed / drifted" is Compliance. "Wrong" is Validity — it needs effectiveness evidence. "Incomplete" is Coverage — a fault happened while every applicable rule was obeyed. Blending them recreates the coarse ambiguity the proofreader method exists to kill.

Compliance
Was the instruction followed? — script, static.
Validity
Did following it help? — needs the whole-story test.
Coverage
Is a needed instruction missing? — residual failure despite full compliance.

03Every bolt has its own score

Not one blended number that hides the loose bolt. Each instruction carries its own scorecard.

Compliance(instruction i) = FOLLOWED / (TRIGGERED and CHECKABLE) · per instruction, with vs without the cookbook Compliance-lift = followed-rate(cookbook on) − followed-rate(off) · does loading the handbook actually change behaviour? Deterministic coverage = instructions with a certified checker / all instructions

Scores roll up — instruction → chapter → book — but stay drillable to the single line. That panel of per-instruction fault codes is the diagnostic: it tells you which lines are honoured, which drift, and which the model ignores entirely.

04A broken bolt gets replaced with a higher-scoring one

The scores don't just diagnose — they let you improve, and prove the improvement.

low-scoring instruction→ rewrite the wording→ test new vs old in isolation→ higher score?→ swap it in

An instruction that's ignored, drifts, is ambiguous, or is followed-but-useless becomes a candidate. Rewrite it, A/B the new wording against the old on the very check that failed, and if it scores higher — more consistently followed, clearer, more effective — replace it. The cookbook stops being a fixed book and becomes a self-improving machine, one measured instruction at a time.

05Phase 2 — Judge the whole story

Clean writing is necessary, not sufficient. Once the bolts pass, run the machine.

The whole-campaign, all-or-nothing effectiveness test from the earlier debate isn't wrong — it's the final integration test, not the first diagnostic. And attribution is two-stage: first, did the cookbook change compliance with instruction X? then, did that changed compliance change a real outcome? If compliance rises but outcomes don't, the instruction is dead weight or its claimed mechanism is wrong. Some instructions only pay off in combination, so line-level tests are checked, then chapter-level interaction tests, then the whole book.

06Scaling — and the honest limits

The checker battery stays constant; only the breadth grows.

  • Unit: instruction → chapter → book (integration). Don't widen a level until the smaller unit's UNKNOWNs are exhausted.
  • Breadth: one author proves feasibility → a group reproduces the same fault codes on frozen bundles → many sites/agents → worldwide independent replication of a published spec. Never a "worldwide vote of AI opinions."
What this cannot do (stated plainly). ≈10% of instructions aren't checkable today (attitudinal, hidden, or counterfactual). Compliance is not value — a perfectly-obeyed useless rule scores high on Phase 1 and zero on Phase 2. And the standing danger is compliance theatre: optimising the cookbook for what's easy to instrument while the real work gets worse. Guardrails: keep the three ledgers separate, treat a pass as "no violation detected" (not "true"), keep UNKNOWNs visible, and never publish a single number without its full fault-code table and its observability denominator.

Recommended first bolt to torque

Build the registry and the Class-A mechanical checkers first — worktree/branch hygiene, clean tree, no orphan branches, no leftover processes, co-author trailers, never-touch-BASE_URL. They're free, instant, un-gameable, and run on every session today. Get a real per-instruction compliance panel for the ~25% that needs no instrumentation, exhaust it, then add the Class-B probes. Small, fast, static — exactly the check-engine panel.
